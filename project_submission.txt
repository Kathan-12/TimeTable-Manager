I.      Brief Description of the Project
     The Time Table Generator System automatically generates a valid and conflict-free university
     timetable. It assigns courses to available time slots and classrooms while satisfying all hard
     constraints such as faculty availability, room capacity, batch scheduling conflicts, and course
     duration requirements.
     The final output includes:
           Batch-wise timetable
           Faculty-wise timetable
           Room allocation schedule
           List of unresolved conflicts (if any)


II.     Process Model Chosen and Justification

            Process Model: Agile (Scrum)

            Justification:
            The Agile model is selected for the Time Table Generator project due to its iterative and
            flexible approach. Since timetable scheduling involves multiple constraints and possible
            refinements, Agile allows requirements and logic to evolve during development.
                 Supports changing requirements during constraint implementation.
                 Enables incremental development through sprints (input module, scheduling
                     logic, reports).
                 Allows continuous feedback and improvement after each iteration.
                 Helps in early risk detection and systematic issue resolution.
                 Encourages team collaboration and transparency.
                 Ensures better quality output through continuous testing.
            Thus, Agile is suitable for developing a constraint-based system like the Time Table
            Generator.

 Algorithm:

   1. Initialize empty timetable
   2. Sort courses (labs first)
   3. For each course:
     ○ Try all combinations
     ○ Check hard constraints
     ○ Assign if valid

   4. If assignment fails:
     ○ Backtrack

   5. Apply soft constraint optimization

VIII.   Programs and the logics applied
        The timetable generator is designed as a structured scheduling system that automatically
        creates conflict-free class timetables for the university. The core logic of the system is built
        around Constraint Satisfaction Problem (CSP) principles, supported by backtracking, greedy
        initialization, validation rules, and optimization heuristics. This ensures that the timetable is

           ● available time slots


          2. The input is checked through validation modules to ensure that all required information
        is correct and complete.


         3. The scheduling engine attempts to assign each course to an appropriate:
           ● faculty

2. ALGORITHMS AND TECHNIQUES USED


 a) Constraint Satisfaction Problem (CSP)


   The timetable generation problem is treated as a CSP where:
   ● Variables: lectures, time slots, faculty, rooms, batches
   ● Domain: all possible valid combinations
   ● Constraints: rules that ensure valid assignments


   Only assignments satisfying all constraints are accepted.


 b) Backtracking Algorithm


                                       Page 27 of 38
 Backtracking is used when:
 ● a conflict occurs during assignment


 Process:
 ● remove the current assignment
 ● try the next possible option


 This continues until:
 ● a feasible timetable is found OR
 ● all possibilities are exhausted


c) Greedy Allocation


 Used for fast initial placement:
 ● high-priority courses (e.g., labs) assigned first


 Advantage:
 ● reduces future conflicts
 ● builds strong initial solution


d) Validation Checks


 Performed before and during scheduling:


 ● faculty availability
 ● room capacity
 ● room type compatibility
 ● batch availability (no double booking)
